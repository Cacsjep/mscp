How It Works

This plugin lets you configure HTTP requests (similar to Postman) that execute automatically when triggered by Milestone XProtect™ rules. You create Request Folders containing individual HTTP Requests. When a rule fires, the "Execute HTTP Request" action runs on the Event Server. The Rules engine lets you target a specific request, all requests in a folder, or all requests across all folders.

Your custom JSON payload is merged with the Milestone event data, giving the receiving system both your custom fields and the full event context. For GET and DELETE requests, the body section is hidden automatically.

Getting Started

  1. Click HTTP Requests in the tree on the left.
  2. Right-click and select Create New to add a Request Folder.
  3. Expand the folder, right-click it, and select Create New to add an HTTP Request.
  4. Configure the request: method, URL, payload, authentication, and headers.
  5. Click Save to persist the configuration.
  6. Go to Rules and Events > Rules and create a rule with the Execute HTTP Request action. You can target a specific request, all requests in a folder, or all requests.

Rule Action

The plugin registers one rule action: Execute HTTP Request. When creating a rule, the XProtect™ Rules engine lets you choose the scope:

  • Specific request - select an individual HTTP Request item.
  • All in a folder - select a Request Folder; all enabled requests inside it will be executed.
  • All requests - select "ALL HTTP Requests" to execute every enabled request across all folders.

This targeting is handled natively by the Milestone Rules engine - no additional configuration is needed in the plugin.

Authentication

Each request supports the following authentication methods:

  • None - No authentication header is sent.
  • Basic - Sends Authorization: Basic <base64>. Enter credentials as username:password.
  • Bearer - Sends Authorization: Bearer <token>. Enter the raw token value.
  • Digest - Uses HTTP Digest authentication (RFC 7616). Enter credentials as username:password. The client will negotiate the digest challenge automatically.

JSON Payload Examples

POST with user payload + event data (merged)

When "Include Milestone event data" is checked, your JSON payload is merged with the event data. Your keys come first, then Event and Site are appended automatically.

Your payload (configured in the request):

{ "alert_type": "motion", "zone": "parking_lot", "severity": "high" }

Resulting payload sent to the endpoint:

{ "alert_type": "motion", "zone": "parking_lot", "severity": "high", "Event": { "EventHeader": { "ID": "e0d6726f-fa1a-4ade-b1c0-fb01e1a175bf", "Timestamp": "2026-03-09T06:16:58.258Z", "Type": "System Event", "Name": "Motion Detected", "Message": "Motion Detected", "Priority": 1, "CustomTag": "", "Source": { "Name": "Camera 512", "FQID": { "ObjectId": "35f564f6-4af8-4f59-a1b0-3e0eaf88ac04", "Kind": "5135ba21-f1dc-4321-806a-6ce2017343c0" } } } }, "Site": { "ServerHostname": "acs", "AbsoluteUri": "https://acs/" } }

POST with event data only (empty user payload)

Leave the user payload empty and check "Include Milestone event data":

{ "Event": { "EventHeader": { "ID": "eba5eb9a-b31f-4b1f-99ec-3d52258fcbae", "Timestamp": "2026-03-09T06:16:58.935Z", "Type": "System Event", "Name": "Motion Detected", "Message": "Motion Detected", "Priority": 1, "CustomTag": "", "Source": { "Name": "RTSP Source (10.0.0.48) - Channel 1", "FQID": { "ObjectId": "57a5944c-0a55-4e62-bb6c-e87b527fa8cd", "Kind": "5135ba21-f1dc-4321-806a-6ce2017343c0" } } } }, "Site": { "ServerHostname": "acs", "AbsoluteUri": "https://acs/" } }

POST with user payload only (no event data)

Uncheck "Include Milestone event data" to send only your custom JSON:

{ "alert_type": "motion", "zone": "parking_lot", "severity": "high" }

GET request with query parameters

For GET and DELETE requests, the body section is hidden. Use the Query Params tab to add key-value parameters, or include them directly in the URL:

Method: GET URL: https://api.example.com/notify Query Params: type=motion, camera=Camera512 No body is sent for GET/DELETE requests.

Comparison: HTTP Requests Plugin vs. Webhooks

Milestone XProtect™ includes a built-in Webhooks add-on. This plugin provides additional flexibility beyond the built-in feature:

Feature Milestone Webhooks (built-in) HTTP Requests Plugin
HTTP Methods POST only GET, POST, PUT, DELETE, PATCH
Custom JSON payload No - fixed schema Yes - fully customizable, merged with event
Custom HTTP headers No Yes - arbitrary key-value pairs
Authentication OAuth 2.0 (client credentials) Basic, Bearer, Digest
Certificate validation skip No Yes - per request toggle
Configurable timeout No Yes - per request
Postman-like UI No Yes - method, URL, headers, body, auth, test, duplicate
Query parameter editor Manual (in URL) Yes - dedicated key-value editor
Result events No feedback events Yes - fires success/failure events for chaining rules
System Log entries Limited Yes - detailed per-request logging
Payload types JSON (fixed) JSON, form-urlencoded, or none

Architecture

The plugin has two components:

Management Client (Admin UI)

Provides this configuration interface. Lets you create, edit, and delete request folders and individual HTTP requests. Configuration is stored in the Milestone configuration database.

Event Server (Background Plugin)

Runs inside the Milestone Event Server. Listens for configuration changes and executes HTTP requests when triggered by rules. Results are logged to the System Log and fired as analytics events for rule chaining.

Management Client Event Server (Admin UI) (Background Plugin) | | |--- Save Config ---------->| | |--- Load folders + requests | | | [Rule fires] | | |<-- NewEventCommand (trigger) | |--- Execute HTTP request(s) | |--- POST https://api.example.com/ | |--- Log result to System Log | |--- Fire success/failure event |<-- MessageComm -----------| | (execution result) |

Logging & Diagnostics

Milestone System Log

All request executions are logged to the Milestone System Log (Logs > System Log in the Management Client):

  • Request Executed - method, URL, status code, elapsed time
  • Request Failed - method, URL, error message

Event Server Log Files

Detailed logs at:
C:\ProgramData\Milestone\XProtect Event Server\logs\MIPLogs\MIP<date>.log
