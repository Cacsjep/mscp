Internal certificate authority and TLS cert vault for XProtect™ servers and devices
Open source - no warranty, use at own risk Stores certs in MIP item config

What this plugin does

The PKI plugin lets you run a small internal certificate authority for your XProtect™ site without leaving the Management Client. You can issue a Root CA and Intermediate CAs, generate HTTPS / 802.1X / Service certificates signed by those CAs, import existing certificates (PEM, DER, PKCS#12), and export them back out for distribution.

Every certificate (and its private key, where present) lives on the Management Server as a MIP item, which means the entire vault travels with your XProtect™ configuration backups.

Typical first run (Auto setup)

From the Overview node, click Auto setup. This will:

  • Generate a self-signed Root CA named MSCP PKI (yyyy.MM) - Root CA valid for 15 years (skipped if it already exists).
  • Discover every XProtect™ service registered on this site (Recording Server, Mobile, Event, Mgmt) and issue one Service certificate per service signed by the Root CA, valid for 15 years.
  • Fill the SAN list of each service cert with the hostname, FQDN, and registered IPs of that service so the cert is immediately usable.

Re-running Auto setup later in a different month produces a new stamped batch and won't collide with the previous one. Existing certs with the same names are never overwritten.

Deploying issued certs

The Mgmt Client side only stores the certificates - it does not push them into the Windows certificate store on each XProtect™ server. Two ways to deploy:

  • PKI Cert Installer (recommended): the standalone single-file EXE shipped next to PKI.dll (and downloadable from the local download page at http://<mgmt-server>/mscp/). Run it on each XProtect™ server with admin rights, sign in to the Mgmt Server, pick the certs that match the machine, click Install. It places the cert in the right Windows store and grants the service account the private-key ACL.
  • Manual export: select certs in the Overview pane and click Export. PFX includes the private key (with optional password); cert-only formats (PEM / DER / .crt) are also available.

Role permissions

Each PKI folder has its own read permission that admins grant per role under Security > Roles > [Role] > MIP > PKI.

PermissionEffect when granted
Read Root CertificatesSee and read certs in the Root CA folder.
Read Intermediate CertificatesSee and read certs in the Intermediate CA folder.
Read HTTPS CertificatesSee and read certs in the HTTPS folder.
Read 802.1X CertificatesSee and read certs in the 802.1X folder.
Read Service CertificatesSee and read certs in the Service folder.

The Overview node is visible as long as the role has read on at least one folder; the cert list in Overview filters per-row by the same per-folder permissions. The PKI Cert Installer EXE talks to the same REST surface, so the same permissions decide which certs an installer operator can deploy on a given XProtect™ server.

Storage and security notes

  • Cert and key bytes are stored as PKCS#12 (PFX) on the MIP item, base64-encoded. Anyone with read access to the Management Server config can extract them.
  • The plugin does not auto-renew. Watch Remaining in the Overview; generate a fresh cert before expiry and re-deploy with the Cert Installer.
  • Deleting a CA item is blocked while other certs in the vault still reference its thumbprint as their issuer. Delete the dependents (or re-issue them against a different CA) first.
