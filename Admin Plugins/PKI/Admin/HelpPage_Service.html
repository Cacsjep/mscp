TLS certificates for XProtect service-to-service traffic
Used by Server Configurator

What lives here

Service certificates secure server-to-server TLS within an XProtect site: Recording Server <→ Mgmt Server, Mobile Server <→ clients, Event Server, Data Collector, and so on. They are leaf certs signed by a CA from this vault.

Auto setup on the Overview pane discovers every registered XProtect service and issues one Service certificate per service automatically, with hostname / FQDN / IPs filled into the SAN list. That's the recommended starting point; this folder is where you pre-stage extras or rotate one without re-running the full Auto setup.

Required fields

  • Sign with - a CA in this vault.
  • Display name - e.g. MSCP PKI (yyyy.MM) - Recording Server srv-01. The Auto setup convention is MSCP PKI (yyyy.MM) - <Category> <Hostname>.
  • Validity - commonly 1 to 5 years for service certs. Internal services aren't subject to the 397-day browser ceiling.
  • Subject Alternative Names - the hostname, FQDN, and every IP the service is reachable on. Add this machine's hostname and IPs pre-fills these from the local machine; on the management server you'll often need to add the SANs of the remote target manually.

Deployment

Run the PKI Cert Installer on the target XProtect server, sign in to the Mgmt Server, pick the Service cert that matches the machine (the Machine Match column shows whether the SANs line up with the local host), click Install, then open the Server Configurator button on the same window to bind the new cert to the relevant role.
